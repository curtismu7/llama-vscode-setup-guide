Overview

AuthZEN is an OpenID Foundation working group developing a standardized API for authorization — often described as the "OpenID Connect of authorization." It creates a common, interoperable interface between the two core components of modern fine-grained access control:

  • Policy Enforcement Points (PEPs) — the applications and gateways that need to ask "can this principal do this action?"
  • Policy Decision Points (PDPs) — the authorization engines that evaluate policy and return a decision.

Before AuthZEN, every PDP vendor had a proprietary API. Applications were locked to a specific engine. AuthZEN eliminates that coupling by defining a standard JSON-based request/response format that any conforming PEP and PDP can use — regardless of the underlying policy language (OPA/Rego, XACML, ALFA, graph-based models).

The problem it solves

Authorization has historically been "fractured" — either baked into application code, embedded as coarse scopes in OAuth 2.0 tokens, or implemented with bespoke proprietary calls to a single vendor's engine. This makes authorization hard to audit, hard to change, and hard to reuse across services.

AuthZEN moves the decision out of the application and into a dedicated, swappable policy engine — and standardizes the wire protocol so that decision can come from anywhere.

AuthZEN API 1.0

The AuthZEN working group published the AuthZEN API 1.0 specification: a JSON-based HTTP API that standardizes how a PEP asks a PDP for an authorization decision.

Core request structure

A PEP sends a structured access request containing three key elements:

  • Subject — who is making the request (user, service, agent identity)
  • Resource — what is being accessed (object, endpoint, document)
  • Action — what operation is being attempted (read, write, transfer, delete)

The PDP evaluates its policy against these inputs and returns a simple, structured decision: permit or deny, optionally with obligations or contextual advice.

Interoperability in practice

Because the API is standardized, a frontend application (PEP) can switch from an OPA-based engine to a XACML engine or a PingOne Authorize instance without changing any application code — only the PDP endpoint changes. This is the same portability OpenID Connect brought to authentication.

Example: Git-based policy engine

One concrete use case from the working group: a Git-based policy engine (policies stored as code in a repository) exposes an AuthZEN-compliant API. Any AuthZEN-conformant application can query it for decisions without writing engine-specific integration code.

AI Agents & Policy-as-Code

AuthZEN is especially relevant for AI agent systems where authorization decisions need to be dynamic, context-aware, and auditable — far beyond what static OAuth scopes can express.

Why static scopes are not enough for agents

An OAuth access token with write grants broad permission. But an AI agent performing a transfer on a user's behalf needs fine-grained decisions: is this specific transfer, for this amount, to this recipient, at this time, within the user's authorized mandate? That is an authorization question — not an authentication question — and it requires a PDP.

Policy-as-Code

AuthZEN enables a Policy-as-Code approach: policies are written in a human-readable language (Rego, ALFA, Cedar), stored in version control, reviewed like code, and deployed to a PDP. The PEP (the agent or gateway) calls the PDP at runtime for every sensitive decision. Changes to policy are tracked, testable, and reversible.

Integration with this demo

PingOne Authorize acts as an AuthZEN-compatible PDP in this architecture. The BFF (Policy Enforcement Point) can call PingOne Authorize to evaluate whether an agent's proposed action is within the scope of the user's delegated consent — combining RFC 8693 token exchange with fine-grained policy evaluation.

Reducing vendor lock-in

Organizations adopting AuthZEN are not locked to any single authorization vendor. They can run OPA internally for low-latency decisions, use a SaaS PDP for complex entitlement graphs, and still have all their PEPs use the same standard API.
