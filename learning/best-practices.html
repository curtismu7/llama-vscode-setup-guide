Know Your Agents

Know Your Agents

Every AI agent must have a registered identity — its own OAuth 2.0 client with a unique client_id. That identity is used to classify the agent, assign a human sponsor, and control what it can do.

🆔Identify & Classify AI Agents

The Super Banking BFF registers a dedicated agent OAuth client separate from the user client. Every MCP token carries a client_id that identifies the agent — not the user, not the BFF app — so audit logs can always attribute actions to the right principal.

👤Assign Sponsors to Govern Agent Access

In this demo the human customer is the implicit sponsor: the agent can only act when the customer is signed in and the session has a valid may_act claim. Production deployments should register a named human sponsor in the PingOne policy as an additional gate.

🔄Manage Agent Lifecycles

The agent client secret is rotated independently of user credentials. TheAGENT_OAUTH_CLIENT_ID / AGENT_OAUTH_CLIENT_SECRET env vars are the lifecycle handle: rotating them revokes all in-flight agent tokens at the next refresh cycle.

How Super Banking implements this

  • ✅ Dedicated agent OAuth client — AGENT_OAUTH_CLIENT_ID env, oauthService.getAgentClientCredentialsToken()
  • ✅ act claim in MCP token — Every MCP call carries act: { client_id: <agent> } after RFC 8693 exchange
  • ✅ Agent identity store — agentIdentityStore.js — persists agent name/role metadata for the UI
  • ⚠️ Human sponsor binding — Implicit via user session; explicit sponsor field is a P2 backlog item

Detect Agents

Detect Agents

An agentic session looks different from a human session. The app needs to recognise that difference and apply tighter controls — scope restrictions, step-up gates, and audit tagging — when an agent is acting.

🏷️Identify When a Session is Agentic

Super Banking marks every request that triggers an MCP tool call. The RFC 8693 exchange produces a token with an act claim — any downstream service can inspect this claim to know the session is agentic without out-of-band signalling.

🛡️Apply Specific IAM Controls for CUA (Computer-Use Agents)

Agent tool calls go through agentMcpScopePolicy.js which enforces an allow-list of scopes before the token exchange even starts. Scopes not on the allow-list are blocked at the BFF — PingOne never sees the exchange request.

📌Tag Agent Sessions

The MCP WebSocket initialize message carries agentToken (the exchanged MCP token) and userSub (the human principal's PingOne subject) as separate fields — the MCP server can log both to tag every server-side action with its human owner and agent executor.
// What the MCP server receives on every tool call
ws.initialize({
  agentToken:     "<exchanged MCP JWT — not the user token>",
  userSub:        "user-abc123",          // human principal
  correlationId:  "req-xyz789",           // tracing
})
// act claim inside agentToken:
//   { "sub": "user-abc123", "act": { "client_id": "bx-agent" } }

How Super Banking implements this

  • ✅ act claim detection — act present in MCP token whenever AGENT_OAUTH_CLIENT_ID is configured
  • ✅ Agent scope allow-list — agentMcpScopePolicy.js + Config UI toggle per scope
  • ✅ Session tagging — userSub + correlationId passed to mcpRpc on every call
  • ⚠️ CUA-specific IAM controls — Scope policy is in place; PingOne Authorize policy integration is Phase 2

Delegation

Use Delegation, Not Impersonation

Impersonation means the agent obtains the user's own token and pretends to be the user. Delegation means the agent gets its own token that is cryptographically linked back to the user — so every action is attributable to both the user (who authorised it) and the agent (who executed it).

🚫Never Forward the User Token

agentMcpTokenService.js enforces RFC 8693 exchange as a hard requirement. If mcp_resource_uri is not configured the call throws 503 mcp_resource_uri_required — there is no configuration or code path that forwards the raw user access token to the MCP server.

🔗Delegated Tokens with Limited Scope

The MCP token issued after RFC 8693 exchange has:
  • Audience narrowed to mcp_resource_uri (not the banking API)
  • Scope narrowed to only the scopes the specific tool requires
  • An act claim identifying the agent client as the current actor

📋Maintain Clear Accountability Back to the Human Principal

The sub claim in the MCP token is still the user's subject — the human principal. The act.sub or act.client_id is the agent. Together they create a two-level chain: on whose behalf and who is acting.
// RFC 8693 token exchange — BFF → PingOne
grant_type=urn:ietf:params:oauth:grant-type:token-exchange
subject_token=<user access token>           // "on whose behalf"
actor_token=<agent client_credentials JWT>  // "who is acting"
audience=<mcp_resource_uri>
scope=accounts:read

// Resulting MCP token claims:
{
  "sub":    "user-abc123",          // human principal
  "act":    { "client_id": "bx-agent" },  // agent actor
  "aud":    "https://mcp.bxfinance.com",
  "scope":  "accounts:read"
}

How Super Banking implements this

  • ✅ RFC 8693 always enforced — mcp_resource_uri required or 503 thrown; no bypass possible
  • ✅ USE_AGENT_ACTOR removed — Always uses performTokenExchangeWithActor when AGENT_OAUTH_CLIENT_ID set
  • ✅ Subject-only fallback warning — on-behalf-of-warning event in Token Chain when no agent client configured
  • ✅ Audience + scope narrowing — MCP token audience = mcp_resource_uri; scope = tool-specific

Least Privilege

Enforce Least Privilege

The agent should have access to exactly what it needs for the current task — no more. Scope, audience, token lifetime, and human approval gates are all tools for keeping the blast radius small if something goes wrong.

🎛️Limit Agent Access — Scope Allow-List

Every tool in the MCP catalog has a required scope set (e.g. accounts:read). agentMcpScopePolicy.js checks the allow-list from Config UI → Agent MCP scopes before the token exchange runs. If the tool's scope is not enabled, the call is rejected at the BFF — PingOne never processes the exchange.

⏱️Use Short-Lived Tokens

The exchanged MCP token has the lifetime set by PingOne on the token-exchange grant (typically shorter than the user token). The BFF middleware (refreshIfExpiring) proactively refreshes the user token when it is within 5 minutes of expiry — keeping the chain live without user friction.

✋Require Human Approval for High-Risk Actions

Transfers over the configured threshold trigger either:
  • Step-up MFA — PingOne Authorize gate on the transfer API endpoint
  • Transaction Consent Modal — explicit in-UI checkbox approval
Both paths log the approval event and block the agent until the human acts.
// Per-tool scope map in mcpWebSocketClient.js
MCP_TOOL_SCOPES = {
  get_my_accounts:       ['accounts:read'],
  get_my_transactions:   ['transactions:read'],
  create_transfer:       ['accounts:read', 'transactions:write'],
  get_account_balance:   ['accounts:read'],
}

// agentMcpScopePolicy.js — checked before token exchange
if (!isToolPermittedByAgentPolicy(toolScopes, agentAllowedSet)) {
  throw { code: 'agent_mcp_scope_denied', httpStatus: 403 };
}

How Super Banking implements this

  • ✅ Per-tool scope enforcement — agentMcpScopePolicy.js — allow-list configurable in Config UI
  • ✅ Token refresh middleware — refreshIfExpiring applied to /api/mcp, /api/demo-agent, /api/accounts etc.
  • ✅ Step-up auth gate — authorizeGate.js + stepUpGate.js on high-value transaction endpoints
  • ✅ Transaction consent approval — TransactionConsentModal — human checkbox required before agent executes transfer
  • ✅ Upstash re-fetch on stale session — server.js middleware recovers tokens on cold-start to prevent silent privilege escalation

Human in the Loop

Human in the Loop

Fully autonomous agents are a liability for sensitive operations. A human must be able to see what the agent is doing, approve high-impact actions, and have every such approval logged for audit.

📲Prompt Human for Sensitive Operations (CIBA)

CIBA (Client-Initiated Backchannel Authentication) sends an out-of-band approval request to the user's registered device before the agent receives the token it needs to proceed. The agent polls until approved — or times out and halts.

In this demo the CIBA Panel and CIMD Simulator let you step through this flow live.

☑️Transaction Consent Modal

High-value transfers pop up an in-app consent dialog that the customer must tick. The agent cannot complete the transfer until the human has explicitly approved the specific amount and recipient. The consent challenge ID is stored server-side and verified before execution.

📋Log Verifications for Audit

Every approval event — CIBA, step-up MFA, consent modal — is written to the activity log (activityLogger.js) with the user subject, agent client ID, tool name, correlation ID, and timestamp. The admin Activity Logs page surfaces these entries in real time.
// CIBA flow — agent waits for human approval
POST /api/auth/ciba/initiate
  { userId, scope, bindingMessage: "Approve $500 transfer to John" }
→ { auth_req_id }

// Poll until approved:
POST /token { grant_type: urn:openid:params:grant-type:ciba, auth_req_id }
→ authorization_pending  (retry every 5s)
→ tokens returned ✓  (human tapped Approve on their device)

// Delegated Access (RFC 8693 extension):
// After approval, exchange yields a token with:
//   sub = user, act = { client_id: "bx-agent" }
// The transfer executes with that token — audit trail complete.

How Super Banking implements this

  • ✅ CIBA backchannel auth — routes/ciba.js + CIBAPanel.js — full flow including CIMD simulator
  • ✅ Transaction Consent Modal — TransactionConsentModal.js — checkbox approval before high-value agent transfers
  • ✅ Step-up MFA gate — stepUpGate.js — PingOne step-up challenge on protected endpoints
  • ✅ Delegated Access UI — /delegated-access — family member delegation with RFC 8693 Act-as explainer
  • ✅ Audit logging — activityLogger.js logs every approval with userId, agentClientId, correlationId

async function resolveMcpAccessTokenWithEvents(req, tool) {
  const userToken = getSessionBearerForMcp(req);  // from session, not browser
  if (!userToken) return { token: null, tokenEvents: [], userSub: null };
  // …may_act checks, scope narrowing, then:
  const mcpToken = useActor

  return { token: mcpToken, tokenEvents, userSub };
}

IETF Standards

IETF Standards Behind These Practices

All five best practices in this panel are built on peer-reviewed, open IETF standards. Each guardrail maps to one or more active security standards from OAuth, WIMSE, JOSE Working Groups.

  • Practice 1 — Know Your Agents: RFC 8693 (Token Exchange), act claim proposal
  • Practice 2 — Detect Agents: RFC 8693 exchanged tokens carry act claims
  • Practice 3 — Delegation: RFC 8693 core standard, Identity Chaining (ID-JAG) for audit
  • Practice 4 — Least Privilege: RAR (RFC 9396) + WIMSE Workload Identity
  • Practice 5 — Human in the Loop: JAG-IR (JWT Grant Interaction Response) for CIBA

All seven standards co-authored by Ping Identity are covered in depth in the IETF Standards: Agentic Identity panel (see Learn & Explore menu).
