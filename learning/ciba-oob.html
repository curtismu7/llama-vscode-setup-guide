What is CIBA

CIBA (Client-Initiated Backchannel Authentication, OpenID CIBA Core 1.0) decouples the consumption device (where the app runs) from where the user approves — often another device or their email inbox. No browser redirect, no popup. PingOne delivers the approval step by email or push depending on your DaVinci configuration.

The flow (6 steps)

1. App (server) ──POST /bc-authorize──▶ PingOne
   { login_hint: "user@bank.com", scope: "openid write",
     binding_message: "Approve $500 transfer" }

2. PingOne ◀─────────────────────────── auth_req_id returned

3. PingOne ──out-of-band approval────▶ User (channel is your PingOne / DaVinci setup)
   • Email: approval link in inbox  — OR —  • Push: notification on registered device

4. User approves (link in email or tap Approve on device)

5. App polls POST /token (grant=ciba, auth_req_id=...)
   → authorization_pending (repeat every 5s)
   → tokens returned ✓

6. Tokens stored server-side (never sent to browser)
   Tool call / transaction executes with user context

Real HTTP: bc-authorize request & response

POST {issuer}/as/bc-authorize
Content-Type: application/x-www-form-urlencoded
Authorization: Basic <base64(client_id:client_secret)>

scope=openid%20banking%3Awrite
&login_hint=user%40bank.com
&binding_message=Approve%20%24500%20transfer
&acr_values=Multi_factor            (optional step-up)
&client_notification_token=...      (required for ping/push delivery mode)

HTTP/1.1 200 OK
{
  "auth_req_id": "abc123xyz...",
  "expires_in": 300,
  "interval": 5
}

Real HTTP: polling for tokens

POST {issuer}/as/token
Content-Type: application/x-www-form-urlencoded

grant_type=urn:openid:params:grant-type:ciba
&auth_req_id=abc123xyz...
&client_id=...
&client_secret=...

── If user has not yet approved ──
HTTP/1.1 400 Bad Request
{ "error": "authorization_pending" }

── If poll is too fast ──
HTTP/1.1 400 Bad Request
{ "error": "slow_down" }   → increase interval by 5s

── On approval ──
HTTP/1.1 200 OK
{
  "access_token": "...",
  "token_type": "Bearer",
  "id_token": "...",
  "refresh_token": "...",
  "expires_in": 3600
}

── On denial / timeout ──
HTTP/1.1 400 Bad Request
{ "error": "access_denied" | "expired_token" }

Key concepts

  • auth_req_id — a short-lived opaque ID returned by PingOne when POST /bc-authorize succeeds. The server uses it to poll POST /token until the user approves or the request expires.
  • binding_message — the text shown in the approval email or push notification, e.g. "Approve $500 transfer to Savings". Helps the user confirm exactly what they are authorising.
  • login_hint — the user's email address. PingOne resolves this to the target account and sends the approval to the right inbox or device.
  • Poll vs Ping delivery mode — Poll: server calls POST /token every 5 s (or interval seconds). Ping: PingOne calls a client_notification_endpoint when the user approves (requires a publicly reachable callback URL). This demo uses Poll mode.
  • Backend-for-Frontend (BFF) pattern — tokens never reach the browser — tokens are stored in the server-side session. The browser only receives approval status updates via the Backend-for-Frontend (BFF) poll API. XSS cannot steal them.
  • Email vs push — controlled by your PingOne / DaVinci flow, not by this app. Email-only CIBA requires no push-capable MFA device.

When to use CIBA (vs Authorization Code)

  • LLM / agent contexts where a browser redirect would break the flow.
  • Step-up authentication mid-session (high-value transaction) without a page reload.
  • IoT / headless devices that cannot host a redirect URI.
  • Delegated approval — approve on phone while viewing dashboard on desktop.

Sign-in & roles

Two different OAuth sign-ins

Each button on the login page starts a different authorization flow using a different PingOne OAuth client (admin app vs end-user app) — two clients on every platform. On hosted deployments, both client IDs and secrets are pre-configured on the server — visitors do not type them in Application Configuration. On localhost, you configure both apps in the Config UI (LMDB).

👑
Admin sign-in — GET /api/auth/oauth/login
Uses the admin PingOne application (admin redirect URI). After PingOne returns, the browser is redirected to the Admin Dashboard at /admin (or FRONTEND_ADMIN_URL if set). In this demo, new users created through this flow get the admin role in the local user store.
👤
Customer sign-in — GET /api/auth/oauth/user/login
Uses the end-user PingOne application (customer redirect URI). New users get the customer role and sample accounts, and land at /dashboard. Existing users keep their stored role — if you previously signed in via Admin sign-in (which sets role: admin in the demo store), the customer flow will also route you to /admin. To ensure a fresh customer session, log out fully before switching flows.

What admins can do that customers cannot

  • Admin (role === 'admin'): Activity logs, manage users, view all accounts and transactions, Security Settings (thresholds / PingOne Authorize integration), Application Configuration link, full MCP Inspector. Routed to /admin and admin-only routes.
  • Customer (role === 'customer'): Personal dashboard only — own accounts, deposits, transfers, withdrawals within normal policy; MCP Inspector in read-oriented use; no tenant-wide admin screens.

The UI loads the signed-in user from GET /api/auth/oauth/status (admin PingOne app), GET /api/auth/oauth/user/status (end-user app), and GET /api/auth/session (basic auth). The post-login redirect depends on the user's stored role — a user whose record has role: admin is routed to /admin regardless of which OAuth app was used for sign-in.

AI Agent (robot button) vs signing in

The AI Agent panel is not a third identity. It does not log you in by itself. When you are signed in, the Agent calls POST /api/mcp/tool; the API server attaches your current session's OAuth access token and forwards MCP tool calls as you. So the Agent is the same user as the browser session — just a different UI (MCP tools) on top of the same Backend-for-Frontend (BFF) session cookie.

When you are not signed in, the Agent only offers Configure and the two login buttons — it cannot run tools until a session exists.

How we know who you are: the server reads req.session.user for the human. For MCP tool calls, the Backend-for-Frontend (BFF) can optionally issue a delegated token where the agent OAuth client is the actor and you remain the subject (RFC 8693 with actor_token + subject_token) — configure USE_AGENT_ACTOR_FOR_MCP, AGENT_OAUTH_CLIENT_*, and optional PingOne directory provisioning via /api/agent/identity/bootstrap. That is “on behalf of,” not impersonation.

How this demo maps to the diagram

  • Web Browser SPA — React UI; session cookie only (no raw AT/RT in JavaScript).
  • Agent / LLM — demo_agent_service reasoning service (port 3006); BFF calls runReasonLoop() against the external TypeScript agent.
  • MCP gateway — demo_mcp_gateway (port 3005): RFC 7662 introspection, GatewayTokenPolicy scope/audience check, PingOne Authorize decision (PERMIT/DENY/INDETERMINATE). The BFF (demo_api_server, port 3001) routes tool calls to this gateway via callToolViaGateway.
  • MCP egress — BFF performs single RFC 8693 exchange per tool call (mcpToolPipeline.js), then routes to demo_mcp_gateway. On PERMIT, gateway forwards original bearer token unchanged to demo_mcp_server.
  • IDP / AS — PingOne (/authorize, /token, introspection).
  • MCP Server + Tool — demo_mcp_server (port 8080); tools call the Banking API with Bearer + scopes.
  • Resource Server (RS) — Banking REST API (same host as Backend-for-Frontend (BFF) in this deployment).

Phased OAuth — resource indicator (RFC 8707)

Phase 1 — Authorization request
  GET /as/authorize
    ?response_type=code
    &client_id=...
    &redirect_uri=...
    &scope=openid%20banking%3Aread
    &resource=https%3A%2F%2Fapi.example.com%2F   ← binds audience at issuance

Phase 2 — Token request
  POST /as/token
  grant_type=authorization_code
  &code=...
  &resource=https%3A%2F%2Fapi.example.com%2F     ← same resource; AS limits aud

Phase 3 — Resource access
  GET https://api.example.com/accounts
  Authorization: Bearer <AT whose aud = https://api.example.com/>

  RS verifies: aud == its own URL; scope includes required permission.

Token exchange for MCP (RFC 8693)

This demo keeps OAuth tokens on the server. When the MCP layer needs an access token with the right audience for tools or delegation, the Banking Backend-for-Frontend (BFF) calls PingOne's POST …/as/token with grant_type=token-exchange. Below: what happens before and after that call, typical HTTP status codes, and success/error JSON shapes.

Where this runs

The browser never calls PingOne /token for MCP. The Banking API server holds the user's OAuth tokens in the session and, when a tool needs an MCP-scoped access token, performs RFC 8693 Token Exchange (grant_type=urn:ietf:params:oauth:grant-type:token-exchange) against PingOne's POST …/as/token endpoint (same host as your issuer).

Before the exchange — inputs the Backend-for-Frontend (BFF) already has

  • Session cookie — identifies the signed-in user (admin or customer).
  • req.session (server) — access_token / refresh_token / user from the initial OAuth code flow.
  • Optional "on behalf of" path — USE_AGENT_ACTOR_FOR_MCP=true and MCP resource URI set → the Backend-for-Frontend (BFF) may use subject_token (user) + actor_token (agent client-credentials token) so PingOne can mint a delegated token (JWT may include an act claim per your AS policy).

The token request (Backend-for-Frontend (BFF) → PingOne, not visible in browser DevTools)

POST {issuer}/as/token
Content-Type: application/x-www-form-urlencoded

grant_type=urn:ietf:params:oauth:grant-type:token-exchange
client_id=…  &  client_secret=…          (or other client auth per your app)
subject_token=<user access token from session>
subject_token_type=urn:ietf:params:oauth:token-type:access_token
audience=<MCP resource / API audience PingOne expects>
scope=<space-separated scopes for the MCP layer>

# Optional actor path (delegation):
actor_token=<token from agent client credentials>
actor_token_type=urn:ietf:params:oauth:token-type:access_token
# (PingOne policy must allow this exchange.)

After a successful exchange — what the Backend-for-Frontend (BFF) receives

HTTP/1.1 200 OK
Content-Type: application/json

{
  "access_token": "<JWT or opaque string — MCP/WebSocket uses this as Bearer>",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "…",
  "issued_token_type": "urn:ietf:params:oauth:token-type:access_token"
}

The Backend-for-Frontend (BFF) then opens or reuses the WebSocket to demo_mcp_server and sends MCP messages that carry this token. Downstream Banking REST calls still use RS audience/scopes as configured.

HTTP status codes

  • 200 — body is JSON with access_token (and usually expires_in).
  • 400 — invalid grant, wrong token type, audience not allowed, malformed request. Body often: {"error":"invalid_grant","error_description":"…"}
  • 401 — client authentication failed (wrong client_id/secret or auth method).
  • 403 — policy or consent blocks the exchange.
  • 502/503 — Backend-for-Frontend (BFF) could not reach PingOne (network/DNS).

Backend-for-Frontend (BFF) API responses you may see in the browser

  • POST /api/mcp/tool — 200 with MCP tool result JSON; 401 if no session or no usable token; 5xx if MCP server or PingOne is unreachable.
  • GET /api/mcp/inspector/tools — 200 with tools list; 401 if not authenticated.
  • POST /api/mcp/inspector/invoke — same pattern as tool calls.

Error JSON shape (OAuth-style, from PingOne through Backend-for-Frontend (BFF))

{ "error": "invalid_grant" | "invalid_client" | …, "error_description": "human-readable" }

Mental model

  1. User completes OAuth (authorization code) → session has subject token.
  2. User invokes MCP tool → Backend-for-Frontend (BFF) may exchange subject token for MCP-audience token (RFC 8693).
  3. Only after step 2 does the MCP layer see a Bearer suited to the tool/RS chain.

Open the Config page → "MCP Inspector setup" for env snippets that match your URL.

vs Login Flow

CIBA vs Authorization Code (login flow)

📲
CIBA
No browser redirect to PingOne for the approval step. User gets out-of-band approval (email or push per your PingOne config); server polls POST /token with grant_type=ciba. Good for agents and step-up without breaking chat.
🔐
Authorization Code + PKCE
Full browser redirect to PingOne login page, then redirect back with code. Immediate session establishment; standard for SPAs with Backend-for-Frontend (BFF). See Learn → Login Flow in the top bar.

Both are OAuth-family flows at PingOne; this app can use CIBA when enabled for high-value actions and uses Authorization Code for sign-in.

When the AI agent needs user tokens to call the backend API, CIBA replaces the awkward "open this URL in your browser" redirect. The Full stack tab shows how MCP tools, token exchange, and REST APIs connect; here we focus on why CIBA helps chat and agent flows.

PingOne Setup

These steps are done in the PingOne Admin Console — no code changes.

1
Enable CIBA Grant Type

PingOne Admin → Applications → (your app) → Grant Types → enable "urn:openid:params:grant-type:ciba"

2
Set Token Delivery Mode

In the CIBA section: Token Delivery Mode = "Poll" (simplest). Set Auth Request Expiry = 300s.

3
Email-only CIBA = no MFA push required

In DaVinci, configure the CIBA flow to send an approval email (notification template with approve link). Users confirm in their inbox — you do not need PingOne MFA push or a registered authenticator app for that path.

4
Optional — push approval instead of email

If your DaVinci flow delivers CIBA via push to a device, enable Push Notification in your MFA policy and register PingOne / MS Authenticator (or your IdP’s device MFA). Email vs push is chosen in PingOne, not in this app.

5
Set CIBA_ENABLED=true

In your .env or hosted environment variables (Replit Secrets, Vercel, etc.), add: CIBA_ENABLED=true

Environment variables

CIBA_ENABLED=true
CIBA_TOKEN_DELIVERY_MODE=poll
CIBA_BINDING_MESSAGE=Banking App Authentication
CIBA_POLL_INTERVAL_MS=5000
CIBA_AUTH_REQUEST_EXPIRY=300
Azure AD users? If PingOne federates to Azure AD, login_hint (email) still resolves the user. For email-only CIBA, no extra device is required. For a push path, users may already have Microsoft Authenticator via Azure AD — depends on your IdP and DaVinci flow.
