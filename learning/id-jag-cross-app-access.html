What is ID-JAG

ID-JAG is the technical acronym for the Identity Assertion JWT Authorization Grant, an emerging IETF standard. While the IETF draft carries the technical name, the marketing term most often used is Cross-App Access (XAA).

Draft reference: draft-ietf-oauth-identity-assertion-authz-grant — IETF OAuth Working Group, active draft (not yet an RFC). The grant type URI is urn:ietf:params:oauth:grant-type:jwt-bearer (re-using RFC 7523 framing with new semantics).

Why it Exists — the Cross-App Problem

Modern enterprises run many applications, each with its own OAuth Authorization Server or tenant. When a user is already authenticated in App A and an agent, service, or second app (App B) needs to act on their behalf — without sending the user through a browser redirect again — there is no clean standards-based path. The options today are all compromises:

  • Share a long-lived service account token (breaks least-privilege)
  • Forward the user's access token directly (breaks audience isolation)
  • Use RFC 8693 Token Exchange (requires deep AS trust configuration)
  • Re-authenticate the user silently (only works same-AS, breaks cross-tenant)

ID-JAG defines a standardised way for an application holding a user's JWT (an ID token or access token) to present that JWT as an identity assertion to a target Authorization Server and receive a new, audience-scoped token — all server-to-server, no user interaction required.

Marketing vs. Technical Name

XAA — Cross-App Access

The marketing / product term. Used in vendor docs (Okta, PingIdentity, Microsoft Entra). Customer-facing. Focuses on the outcome: seamless access across applications without re-login.

ID-JAG — Identity Assertion JWT Authorization Grant

The technical IETF draft name. Used in standards work and protocol implementation. Focuses on the mechanism: a JWT assertion exchanged at a token endpoint.

Key Actors

Asserting Party (AP)

The application that holds the user's current JWT and presents it as an identity assertion. In an agentic context this is typically the orchestrating agent or BFF.

Identity Provider (IdP / AS)

The Authorization Server at the target application that evaluates the incoming JWT, applies its own policies, and issues a new scoped token. PingOne acts in this role.

Subject

The end user whose identity is being asserted. They are represented in the JWT by their sub claim — they are never redirected.

Relying Party / Resource Server

The API or service that will accept the new token issued by the target AS. It never sees the original assertion JWT.

How It Works

ID-JAG uses the existing OAuth 2.0 token endpoint with a special grant type. The assertion JWT is presented directly — no browser involvement.

Token Request — HTTP Shape

POST /oauth2/token HTTP/1.1
Host: auth.target-app.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=urn%3Aietf%3Aparams%3Aoauth%3Agrant-type%3Ajwt-bearer
&assertion=<JWT signed by asserting party>
&scope=read%3Aaccounts
&client_id=app-b-client
&client_secret=...
The assertion is the JWT carrying the user's identity — typically an OIDC ID token or a purpose-built identity assertion JWT. The target AS must be configured to trust the issuer of this JWT.

The Assertion JWT — Required Claims

ClaimRequiredDescription
issYesIssuer — the asserting party; target AS validates this against its trust list
subYesThe user's unique identifier at the asserting party
audYesMust be the target AS token endpoint URL (prevents token replay to wrong AS)
expYesExpiry — short-lived (seconds to minutes), prevents replay
iatYesIssued-at timestamp
jtiRecommendedJWT ID — uniqueness check prevents replay attacks
email / nameOptionalIdentity hints for account matching at the target AS

End-to-End Flow

1

User logs in to App A (the asserting party)

A normal OIDC Authorization Code + PKCE flow. The user authenticates and App A receives an ID token and access token from AS-A.

2

App A needs to call App B on the user's behalf

An agent, service, or BFF in App A determines that it needs a token for App B. The user is not present — this is a background server-to-server call.

3

App A constructs and signs an identity assertion JWT

The JWT includes the user's sub, sets aud to AS-B's token endpoint, and is signed with App A's private key.

4

App A sends the assertion to AS-B's token endpoint

Uses the urn:ietf:params:oauth:grant-type:jwt-bearer grant type. AS-B looks up App A in its trust registry.

5

AS-B validates the assertion and maps the identity

AS-B verifies the JWT signature against App A's JWKS, checks aud, exp, and jti replay. Then it maps the incoming sub to a local identity using its account-linking configuration.

6

AS-B issues a scoped access token for its own resources

App A can now call App B's APIs with a properly scoped access token. The user's identity is propagated — App B knows who the token is for — but the user was never redirected.

User Browser (logged in once to App A)
↓
App A / BFF builds identity assertion JWT, sends to AS-B token endpoint
↓
AS-B (PingOne) validates assertion → maps sub → issues App-B access token
↓
App A / BFF calls App B API with new access token
↓
App B Resource Server serves response — knows the token's user identity

PingOne SSO

PingOne's SSO capabilities map partially to the ID-JAG/XAA model. Some building blocks are present; native end-to-end ID-JAG is not yet a shipped product feature.

What PingOne Can Do Today

✅ Supported: RFC 7523 JWT Bearer — client authentication. PingOne supports using a signed JWT as a client credential (private_key_jwt). This is used in this demo for the AGENT_OAUTH_CLIENT_ID client. The AS validates the JWT, authenticates the client, and issues a client-credentials token.
✅ Supported: RFC 8693 Token Exchange. PingOne supports token exchange: present a subject token (user access token) and receive a delegated audience-scoped token. This is the primary mechanism used in this demo (BFF → PingOne → MCP access token). See the Token Exchange education panel for details.
✅ Supported: OIDC Federation / trust with external IdPs. PingOne Identity can be configured to accept external OIDC-signed ID tokens as the basis for a login (via Identity Provider connections). This is the closest PingOne feature to "trusting assertions from another AS."
✅ Supported: DaVinci orchestration flows. Ping DaVinci can be configured to accept an inbound JWT from a trusted source and orchestrate a login/user-linking flow without redirecting the user's browser — a configurable approximation of XAA for DaVinci-capable deployments.
⚡ Partial Support: Account linking / identity federation. PingOne supports linking external IdP identities to local accounts. An incoming assertion could trigger account lookup by email or external sub. However this requires a full login flow initiation — not a pure token endpoint assertion.

ID-JAG Implementation Pattern in PingOne

The closest approximation achievable with current PingOne capabilities requires combining two features:

  1. Configure App A as a trusted OIDC Provider in PingOne via External Identity Providers. Upload App A's JWKS so PingOne can verify its JWT signatures.
  2. Use the OIDC login flow with prompt=none — send the user's existing id_token as a login_hint_token to hint the identity. PingOne completes the flow silently if the identity maps.
  3. RFC 8693 Token Exchange after the above to narrow the audience to the specific resource server (MCP, banking API, etc.).
This multi-step approximation requires user-session state and cannot be done purely token-endpoint-to-token-endpoint. It is not the same as a native ID-JAG flow, which is a single POST to a token endpoint with no browser redirect at all.

⚠️ Limitations

ID-JAG is an active IETF draft — not yet an RFC. These limitations fall into three buckets: standard gaps, PingOne product gaps, and this demo's gaps.

Standard / Protocol Gaps (IETF Draft Status)

⚠️ Not Supported: The grant type urn:ietf:params:oauth:grant-type:jwt-bearer is shared with RFC 7523 (JWT Bearer for client auth and grants). The ID-JAG draft re-uses this URI with new semantics. Until the draft reaches RFC status, implementors can face ambiguity about which semantics an AS applies when it receives this grant type.
⚠️ Not Supported: No finalised account-linking mandate. The draft does not mandate how the target AS must map an incoming sub from a foreign issuer to a local identity. Each AS implementation (PingOne, Okta, Entra) makes its own choices — creating interoperability gaps between vendors.
⚠️ Not Supported: No standard replay-prevention mandate. The draft recommends jti-based replay detection but does not require it. AS implementations vary in how strictly they enforce this.
The IETF OAuth WG expects the draft to progress to Proposed Standard in 2025–2026. Until then, treat ID-JAG as experimental in production systems.

PingOne Product Gaps

⚠️ Not Supported: No native ID-JAG token endpoint support. PingOne does not expose a token endpoint that accepts a raw JWT assertion as the sole credential and issues an access token in a single round-trip without a prior SSO session. There is no PingOne feature called "Identity Assertion Grant" or "Cross-App Access" in the admin console as of this writing.
⚠️ Not Supported: JWT Bearer assertion grant (RFC 7523 §2.1) not supported as a standalone user grant. PingOne supports private_key_jwt for client authentication, but not the companion grant type where the JWT asserts a user identity to acquire a user token without a prior session.
⚠️ Not Supported: Cross-environment ID-JAG not supported. If App A and App B live in different PingOne environments (different tenants or customer deployments), there is no out-of-the-box token-endpoint assertion mechanism. RFC 8693 token exchange works within a single environment only.
⚡ Partial Support: DaVinci partial support. A DaVinci flow can be authored to accept an inbound signed JWT, look up a user, and return tokens. This is not a standards-compliant ID-JAG implementation — it is a custom adaptation of DaVinci's capabilities. It works if your customer has DaVinci licensed and is willing to build and maintain the flow.
⚡ Partial Support: External IdP federation + silent OIDC. The approximation described in the PingOne tab works but requires browser involvement (prompt=none redirect). This is not server-to-server and breaks in agent contexts where there is no browser session.

This Demo's Gaps

⚠️ Not Supported: Demo uses RFC 8693 Token Exchange, not ID-JAG. The BFF's exchangeTokenRfc8693() function calls PingOne's token exchange endpoint — this is not an ID-JAG grant. The demo cannot demonstrate a native ID-JAG flow because PingOne does not support it natively. The flow is: user access token (subject) + agent client credentials (actor) → MCP access token. This is RFC 8693, not Cross-App Access.
⚠️ Not Supported: No cross-app boundary in the demo. All token exchanges occur within a single PingOne environment. True XAA requires crossing an application or tenant boundary. The demo's single-environment design means it cannot illustrate the multi-tenant or cross-app aspects of XAA.
⚠️ Not Supported: No JWKS endpoint on the BFF. For App A to act as an asserting party, it needs to expose a JWKS endpoint and sign JWTs with a private key. The demo's BFF banking_api_server does not generate or expose key pairs for assertion signing. Adding this would require RSA/EC key generation at startup, a /jwks.json route, and session-bound JWT minting.
Roadmap note: A future phase could add a minimal ID-JAG simulation: configure a second PingOne application as "App B", add a /.well-known/jwks.json route to the BFF, and demonstrate the assertion JWT token endpoint call. This would require DaVinci or a custom PingOne policy to evaluate the assertion on the receiving side.

vs RFC 8693

ID-JAG and RFC 8693 Token Exchange are complementary but distinct. Understanding the difference is critical for recommending the right solution.

When to Recommend Each

Use RFC 8693 when…

  • All apps share a single PingOne environment
  • Need delegation chain (act / may_act)
  • Narrowing scope or audience within AS
  • PingOne is available as the token exchange endpoint

Use ID-JAG / XAA when…

  • Apps are in different tenants / organisations
  • No shared session store between AS-A and AS-B
  • Need portable identity assertion across vendor AS
  • Target AS supports the jwt-bearer grant type with XAA semantics
In practice, most PingOne customers today should use RFC 8693 Token Exchange for agentic cross-service access. ID-JAG becomes relevant when a customer has a multi-cloud or multi-vendor IdP landscape where the agent's AS and the resource's AS are different systems with no shared token base.

Relationship to Other Standards

RFC 7523 — JWT Bearer Grant draft ↗

Published RFC that defines using a JWT as a grant credential (§2.1) and as client authentication (§2.2). ID-JAG re-uses the grant type URI from RFC 7523 §2.1 but extends the semantics for cross-app identity assertion. RFC 7523 alone is not XAA.

draft-ietf-oauth-identity-chaining draft ↗

A related IETF draft describing how to propagate user identity across trust domains in an authorization chain. ID-JAG and Identity Chaining are complementary: ID-JAG provides the assertion mechanism; Identity Chaining defines the trust model and chain structure.

OpenID Connect for Identity Assurance (OIDC4IDA) draft ↗

OID4IDA defines structured claims for verified identity. ID-JAG assertions may carry OID4IDA-structured claims so the receiving AS can make trust decisions based on the level and method of identity verification at the asserting party.
