Overview

IETF Standards Stack for Agentic Identity

Ping Identity co-authors seven active IETF standards governing AI agent identity, delegation, and trust. Each standard addresses one or more of IDC's five AI governance guardrails.

IDC AI Governance Guardrails — How each is implemented in this demo:

#IDC GuardrailStandardsDemo
01Verifiable credentials for repeatable, portable identity proofing
Maturity: Near-RFC / Final		SD-JWT VC + OID4VCI + OID4VP❌
02Delegated authorization with full audit trail across agent chains
Maturity: Very High		Identity Chaining + ID-JAG + RFC 8693 Token Exchange✅
03Explainability logs — every action traceable to its authorization decision
Maturity: Current / High		RFC 8693 nested act claims + Identity Chaining audit chain✅
04Real-time workload attestation and trust across system boundaries

RFC7523bis

RFC7523bis — JWT Client Auth Update

RFC7523bis

What it defines: A fix for an audience-confusion security vulnerability affecting the aud claim across RFC 7521, RFC 7522, RFC 7523, and RFC 9126 (PAR) — requires the authorization server's issuer identifier to be the sole audience value on tokens sent to it, and requires the AS to reject any token that doesn't contain it

Working Group: OAuth WG

Maturity:
High

Ping Role: Co-author

Current Demo Implementation:
✅ ✅ Full

Gap: No current gap — used in agent OAuth client flows

IETF Draft ↗

How the demo uses RFC7523bis: The banking BFF uses JWT client authentication to obtain agent tokens. The agent's OAuth client is identified by client_id, authenticated with a JWT signed by the agent's private key. This ensures the agent's identity is cryptographically bound to the token request.

Identity Chaining

Identity Chaining + ID-JAG

Identity Chaining

What it defines: Cross-domain delegation + Identity Assertion JWT Grant (ID-JAG) — allows a token holder to request a new token on behalf of another principal, with the entire chain cryptographically linked

Working Group: OAuth WG

Maturity:
Very High

Ping Role: Co-author

Current Demo Implementation:
⚠️ ⚠️ Partial

Gap: act claims form a chain, but cross-domain delegation is not demonstrated

IETF Draft ↗

How the demo uses Identity Chaining: When the bankingAgent makes a token exchange request via RFC 8693, the issued MCP token carries an act claim showing the full delegation chain: human user → agent → MCP server. This enables audit trails and accountability across the entire agent workflow.

JAG-IR

JAG-IR — JWT Grant Interaction Response

JAG-IR

What it defines: An extension to the JWT Authorization Grant (RFC 7523) that lets an authorization server signal that user interaction is required to complete a request — returning an interaction URI and polling interval instead of an access token or error, enabling human-in-the-loop (HITL) approval flows for agent operations.

Working Group: Individual submission (OAuth WG adjacent — not yet WG-adopted)

Maturity:
Early

Ping Role: Co-author

Current Demo Implementation:
✅ ✅ Full

Gap: None — CIBA HITL flow is fully implemented

IETF Draft ↗

How the demo uses JAG-IR: High-value operations (transfers, sensitive account changes) trigger out-of-band approval via CIBA (Client-Initiated Backchannel Authentication). The agent polls until the human approves or denies. Only after approval does the agent receive the delegated token to complete the action. All approvals are logged for audit.

AIMS

AIMS — AI Agent Auth Framework

AI Agent Auth (AIMS)

What it defines: A comprehensive agent authentication and authorization framework combining WIMSE workload identity with OAuth 2.0 token flows, scopes, and delegated access patterns.

Working Group: Individual submission (presented in IETF DISPATCH; not yet WG-adopted)

Maturity:
Early

Ping Role: Co-author

Current Demo Implementation:
⚠️ ⚠️ Partial

Gap: Agent identity via OAuth is implemented; WIMSE workload identity binding is not

IETF Draft ↗

How the demo uses AIMS: Each AI agent has its own registered OAuth 2.0 client with a unique client_id and secret. The agent authenticates to obtain tokens, which are narrowly scoped (only the tools it needs). Token exchanges preserve the delegation chain so every action is attributable to the correct agent and human principal.

WIMSE

WIMSE — Workload Identity

WIMSE

What it defines: Workload Identity Made Easy — a standard for cryptographically binding workload identity (e.g., an MCP server, a microservice, an agent) to its OAuth tokens, eliminating reliance on secrets stored in config files.

Working Group: WIMSE WG

Maturity:
Medium-High

Ping Role: Co-author

Current Demo Implementation:
❌ ❌ Not implemented

Gap: Current demo uses environment-variable client secrets. WIMSE workload attestation (short-lived, cryptographically bound credentials) is not yet a native PingOne feature — it requires a SPIFFE/SPIRE or equivalent sidecar alongside PingOne as the authorization server. The standard is still maturing in the IETF WIMSE WG.

IETF Draft ↗

PingOne role: PingOne can act as the authorization server for workload tokens, but the WIMSE-specific workload attestation binding — where a process proves its identity without any stored secret — is outside PingOne's current product surface. A SPIFFE/SPIRE integration would attest the workload and exchange the SVID for a PingOne-issued token.

Roadmap: As the WIMSE WG finalizes the spec, PingOne is expected to add native workload attestation support. When available, static secrets between the banking BFF and MCP server will be replaced with short-lived, cryptographically bound workload tokens — closing IDC Guardrail 04 at full fidelity.

SD-JWT VC

SD-JWT VC — Selective Disclosure Credentials

SD-JWT VC

What it defines: Selective Disclosure JWTs for Verifiable Credentials — allows a credential issuer to issue a single JWT that can be presented to different verifiers, selectively revealing only the fields each verifier needs.

Working Group: OAuth WG

Maturity:
Very High

Ping Role: Co-author

Current Demo Implementation:
❌ ❌ Not implemented

Gap: This demo issues standard PingOne OAuth tokens. SD-JWT VC requires a credential issuance flow (OID4VCI), a wallet to hold the credential, and a presentation protocol (OID4VP) — all outside the current demo scope.

IETF Draft ↗

PingOne role: PingOne Credentials (a dedicated product) fully supports SD-JWT VC issuance via OID4VCI and presentation via OID4VP. Adding it to this demo would require a credential issuer configuration, a wallet application or library, and an OID4VP verification step at the banking API.

Roadmap: Future phases will issue agent credentials as SD-JWT VCs via PingOne Credentials, enabling agents to prove attributes (e.g., delegation level, scopes) to downstream systems with selective disclosure. This fulfills IDC Guardrail 01 at full fidelity.

PQ/T JOSE

PQ/T JOSE — Post-Quantum Cryptography

PQ/T JOSE

What it defines: IETF registrations for Post-Quantum and Hybrid cryptographic algorithms in JOSE (JSON Object Signing and Encryption), enabling JWTs signed with quantum-resistant algorithms.

Working Group: JOSE WG

Maturity:
Medium

Ping Role: Co-author

Current Demo Implementation:
❌ ❌ Not implemented

Gap: PingOne currently signs tokens with RS256/ES256. Post-quantum JOSE algorithm registrations (ML-DSA, SLH-DSA) are still being finalized by the JOSE WG — no commercial identity provider supports PQ/T JOSE yet.

IETF Draft ↗

Context: NIST finalized the first post-quantum cryptography algorithms in 2024 (ML-KEM, ML-DSA, SLH-DSA). The IETF JOSE WG is actively registering these for use in JWTs and JOSE objects, but the draft is not yet finalized. Adoption by token issuers — including PingOne — is a 2026–2027 horizon item.

Roadmap: When PingOne adds PQ/T JOSE signing support, this demo will demonstrate hybrid signing (e.g., ES256+ML-DSA-65), protecting agent tokens against cryptographically relevant quantum computers (CRQC) and fulfilling IDC Guardrail 05.
