RFC Foundations

Intent-based authorization in AI financial systems is built on a foundation of open RFCs and IETF standards. These standards define how delegated authority, scopes, and user intent travel through multi-hop authentication chains.

RFC 8693: OAuth 2.0 Token Exchange

The cornerstone standard for delegated AI agent authorization. RFC 8693 defines two patterns:

  • Impersonation — Subject token alone; the intermediate service is invisible in the new token. Weaker pattern, used for simple proxies.
  • Delegation — Subject token + actor token; the new token includes an act claim recording who is acting on behalf of whom. Enables full audit trail and policy enforcement.

RFC 8693 — OAuth 2.0 Token Exchange (ietf.org)

Key JWT Claims for Delegation

  • act — The actor (AI agent) performing the action. Nested act claims enable multi-hop delegation: user → agent1 → agent2 → service.
  • may_act — Who is authorized to act. Enforced at the authorization server as a policy constraint. Enables revocation: remove an agent from may_act, all its tokens become invalid.

RFC 9396: Rich Authorization Requests (RAR)

Defines authorization_details — a structured way to encode intent in the authorization request itself. Instead of broad scopes (read, write), RAR lets the client request specific actions with context:

{
  authorization_details: [
    {
      type: "payment",
      actions: ["initiate", "approve"],
      amount: "2000.50",
      currency: "USD",
      recipient: "merchant-123",
      time_window: "2026-05-31T00:00:00Z/2026-06-30T23:59:59Z"
    }
  ]
}

The resulting token carries verifiable intent that downstream services can inspect without trusting the agent's assertions.

RFC 9396 — Rich Authorization Requests (ietf.org)

Identity Chaining & Act Claims (Drafts)

Emerging drafts (e.g., oauth-identity-chaining) define how nestedact claims chain across multiple agents. Each hop adds an act record:

// Token at agent2
{
  "sub": "user-123",
  "act": {
    "sub": "agent1-id",
    "act": {
      "sub": "bff-service"
    }
  }
}

This enables downstream services to verify the full delegation chain and enforce policies per hop.

Industry Standards

The banking and payments industry is converging on intent-based authorization standards for agentic AI. Three major initiatives are shaping 2025–2026:

Mastercard Verifiable Intent (2026)

A payment network authentication layer that establishes and documents the human authorization behind AI agent transactions. Mastercard Verifiable Intent:

  • Issues tokenized spending mandates cryptographically linked to user-approved intent (what the agent can do, up to what amount, for how long).
  • Enforces selective disclosure — each party in a transaction receives only the minimum information needed (merchant doesn't see the user's spending caps, only proof that this transaction is authorized).
  • Integrates with risk scoring — the mandate itself carries a risk classification (low / medium / high) used to decide whether step-up MFA is needed.

Mastercard Verifiable Intent (2026)

Google Agent Payments Protocol (AP2)

An open standard for AI agent commerce, announced September 2025 with 60+ launch partners (American Express, PayPal, Coinbase, Salesforce, Worldpay, and others). AP2 defines every agent purchase as three W3C Verifiable Credentials:

  1. Intent Mandate — what the user wants (e.g., "Book a flight under $800 on Delta or United").
  2. Cart Mandate — what the agent assembled (specific flights, prices, taxes).
  3. Payment Mandate — what the merchant or payment network will charge (final amount, fees, settlement info).

Each mandate is cryptographically signed, so any intermediary can verify the chain of intent → assembled cart → final payment without trusting the agent or merchant.

AP2 Protocol Explained (Google)

Know Your Agent (KYA) Framework (2026)

A cross-industry framework for establishing and verifying AI agent identity, authority, and auditability. KYA combines four pillars:

  • Agent identity passport — Verifiable proof that this agent is legitimate (not a spoofed bot). Uses DID, signed certificates, or cryptographic key binding.
  • Authority binding — Explicit proof that the user authorized this specific agent to act on their behalf (consent record, mandate, signed grant).
  • Runtime controls — Policy constraints enforced during execution (spending caps, rate limits, category restrictions, time windows).
  • Tamper-evident audit — Cryptographic proof of every action taken, timestamped and signed, enabling dispute resolution and regulatory compliance.

Implemented by: Experian Agent Trust, Worldpay + Trulioo, and others.

Know Your Agent: An Identity Framework (Trulioo)

FIDO Alliance — Agentic Commerce Standards (April 2026)

The FIDO Alliance announced a new Agentic Authentication Technical Working Group to develop interoperable standards for agent-initiated transactions and trusted interactions. Drawing from AP2 (Google) and Verifiable Intent (Mastercard) contributions, FIDO is building the cryptographic foundations for agent identity and authorization across industries.

Confidence Scoring

Intent-based authorization gates actions on three independent confidence dimensions. Understanding each is key to designing safe agent boundaries.

Three Dimensions of Confidence

1. Intent Confidence (0–1 score)

Question: How certain is the agent that it correctly understood what the user wants?

  • Source: NLP model confidence, semantic similarity, user history patterns.
  • High (≥0.9): Clear user request, unambiguous intent, strong semantic match.
  • Medium (0.7–0.89): Reasonable interpretation, some ambiguity, low-confidence NLP.
  • Low (<0.7): Unclear request, competing interpretations, noisy input.

2. Risk Confidence (0–1 score)

Question: What is the fraud, compliance, or operational risk of this specific transaction?

  • Factors: Amount, merchant, geo-location, time-of-day, device, account history.
  • Sources: Risk models, rule engines, continuous ML learning on transactions.
  • High confidence = low risk: Normal amount, familiar merchant, recognized device.
  • Low confidence = high risk: Unusual amount, new merchant, impossible location, suspicious patterns.

3. Authority Confidence (0–1 score)

Question: Did the user actually authorize this agent to do this?

  • Source: KYA binding, mandate verification, cryptographic proof of consent.
  • High: Explicit recent consent (mandate), signed grant, user enrolled the agent.
  • Medium: Consent from general OAuth scope, inferred from previous actions.
  • Low: Stale consent, no explicit mandate, agent is new to the user.

Threshold Configuration

Different institutions set different thresholds based on their risk tolerance:

  • Conservative (banks, brokerages): Intent ≥ 0.85, require consent for any Risk < 0.8.
  • Moderate (fintech, e-commerce): Intent ≥ 0.7, require consent for Risk < 0.5.
  • Permissive (low-value actions): Intent ≥ 0.5, permit if Risk < 0.3.

Our Implementation

This banking demo implements intent-based authorization using standard OAuth primitives (RFC 8693 token exchange) and configurable authorization rules. Here is how each industry concept maps to demo code:

Code Architecture

  • Feature flag: ff_intent_authorization_enabled (configStore, default: false). Allows disabling the feature without code changes.
  • Intent extraction: Tool name → intent label (e.g., create_transfer → transfer). Happens in agentInvokeRoute.js after agent execution.
  • Authorization decision: intentAuthService.evaluateIntentAuthorization() — evaluates intent + confidence + amount against configurable rules.
  • HTTP responses: 200 (permit), 428 (require consent), 403 (deny).
  • Audit logging: Every decision logged via appEventService.logEvent('intent_auth'). Visible in admin event feed (/api/app-events).

What Changed: Three Critical Gaps Fixed

✅ Real Confidence Scoring: Heuristic NL parser now returns confidence 0–1 based on match quality:
  • Exact phrase match: 0.95 (e.g., "transfer $100 from checking to savings")
  • Partial params: 0.75–0.85 (e.g., "show balance")
  • Single keyword: 0.60–0.65 (e.g., just "transfer")
  • Unknown: 0.30
✅ Pre-Execution Intent Gating: Intent is now extracted from the prompt BEFORE the agent runs tools. This means:
  • Denied intents reject immediately (403) — agent never runs
  • Consent-required intents gate before execution (428)
  • No write operations execute until intent is approved
✅ Risk & Authority Dimensions: Intent gate now evaluates three independent dimensions:
  • Intent Confidence: NL parse accuracy (0–1)
  • Risk Score: Fraud indicators — amount, velocity, account type (0–1)
  • Authority Binding: Is the agent authorized for this user/scope? (0–1)
Decision Logic: High confidence + low risk + full authority = permit. Borderline cases = consent required.

Enhancements

The current implementation covers the core pattern. The next level of maturity requires five enhancements to align fully with industry standards:

1. Real Confidence Scores from LLM

Current state: Hardcoded constants (0.95 / 0.7).

Enhancement: LangGraph agent and heuristic path both return an explicitconfidence field in the response payload. The value should reflect the model's actual confidence in the intent extraction (0–1).

Benefit: The intent_min_confidence threshold becomes a real gate tied to model output, enabling dynamic escalation based on what the model actually knows.

// Agent response (ideal)
{
  "message": "...",
  "intent": "transfer",
  "confidence": 0.87,  // from model
  "toolsCalled": ["create_transfer"],
  "amount": 1500
}

2. W3C Verifiable Credentials for Mandates

Current state: Configurable rules in configStore (comma-separated strings).

Enhancement: Encode user spending mandates as W3C Verifiable Credentials. Each mandate is a signed JSON-LD document specifying:

  • User identity (holder)
  • Agent identity (authorized actor)
  • Scope (e.g., transfer up to $5,000/month to trusted merchants only)
  • Expiration (e.g., valid until 2026-12-31)
  • Signature (issuer's cryptographic proof)

Benefit: Mandates are portable, portable across institutions, and verifiable offline. Matches AP2 and Mastercard Verifiable Intent patterns.

3. Know Your Agent (KYA) Identity Binding

Current state: No agent identity tracking. Decisions don't record which agent acted.

Enhancement: Register each agent instance with a verifiable identity (DID, signed certificate, or cryptographic key binding). Every intent auth decision logs the agent's identity. Enables agent-level revocation and audit trails.

// Intent auth decision (enhanced)
{
  agent_id: "did:key:z6MkhaXgBZDvotDkL5257faWLpa6CBu1...",
  agent_name: "LangChain-Agent-v1.2",
  user_id: "user-123",
  intent: "transfer",
  confidence: 0.87,
  decision: "permit"
}

Benefit: Full audit trail per agent, enables KYA compliance, supports agent lifecycle management (revoke, upgrade, sunset).

4. Explicit Amount Extraction from Agent Response

Current state: Amount inferred from agentResponse.amount || agentResponse.hitl_threshold_usd.

Enhancement: Agent explicitly returns transaction_amount in every response. No guessing, no fallbacks — the amount is a first-class field.

Benefit: Removes ambiguity, enables precise amount-based gating, makes audit trails clear.

5. Multi-Hop Delegation Audit Trail

Current state: Single-hop delegation only (user → agent → service). Theact claim records the immediate agent, not the full chain if the agent spawns sub-agents.

Enhancement: When an agent spawns a sub-agent, each hop adds a nestedact claim. The token carries the full delegation chain.

// Multi-hop delegation (user → agent1 → agent2 → service)
{
  "sub": "user-123",
  "act": {
    "sub": "agent1-id",
    "act": {
      "sub": "agent2-id"
    }
  }
}

Benefit: Supports agentic teams, enables per-hop policies, full compliance and audit visibility.

Roadmap: These enhancements are sequenced by dependency and complexity. Start with real confidence scores + amount extraction, then KYA binding, then W3C VCs, then multi-hop chains.
