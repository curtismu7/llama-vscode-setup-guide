What & Why

What is token introspection? (RFC 7662)

Token introspection is a server-to-server call where a resource server or MCP server asks the authorization server: "Is this token active, and what are its claims?" PingOne returns active: true/false and the token's claim set.

Unlike JWKS-based local validation, introspection always reflects the current state at the AS — including revoked tokens, updated scopes, or act claims that may not be in the token's JWT body.

POST /as/introspect — request

POST {issuer}/as/introspect
Content-Type: application/x-www-form-urlencoded
Authorization: Basic <base64(client_id:client_secret)>

token=<access-token-to-check>
&token_type_hint=access_token

Response — active token

HTTP/1.1 200 OK
Content-Type: application/json

{
  "active": true,
  "sub": "user-pingone-id",
  "aud": "https://api.example.com/",      ← resource server audience
  "iss": "https://auth.pingone.com/.../as",
  "client_id": "banking-bff-client-id",
  "scope": "openid read",
  "exp": 1712345678,
  "iat": 1712342078,
  "jti": "unique-token-id",
  "act": {                                 ← delegation claim (if token exchange used)
    "sub": "agent-client-id"
  }
}

Response — inactive / revoked / expired token

HTTP/1.1 200 OK
Content-Type: application/json

{
  "active": false
}
// No other claims are returned for inactive tokens.
// Treat this as: reject the request with 401.

Error responses

── Bad client credentials ──
HTTP/1.1 401 Unauthorized
{ "error": "invalid_client", "error_description": "Client authentication failed" }

── Malformed request ──
HTTP/1.1 400 Bad Request
{ "error": "invalid_request", "error_description": "Missing token parameter" }

Introspection vs JWKS local validation

Feature                  Introspection (RFC 7662)   JWKS local (JWT signature)
─────────────────────────────────────────────────────────────────────────────
Network call per request YES                        NO (cache public keys)
Reflects revocation      YES (always current)       NO (only until exp)
Works with opaque tokens YES                        NO (needs JWT structure)
act / delegation claims  YES (AS adds them)         Only if in JWT body
Latency                  ~50-200ms round trip       <1ms (local verify)
Best for                 MCP / proxy validation,    High-throughput RS,
                         opaque tokens, act claims  JWT access tokens

In this demo:
  Introspection — user access token at login + every MCP tool call
  JWKS verify   — agent CC tokens + all exchanged/delegated tokens

When to use introspection

  • The token is opaque (not a JWT) — you cannot decode it locally.
  • You need delegation claims (act) that may not be in the JWT body.
  • You need to check revocation in real time (e.g. after a user's session is terminated).
  • The resource server does not want to cache JWKS or manage key rotation.

Where introspection fits in this demo

Introspection is used in two places in the token chain:

  • At login — the BFF calls PingOne's RFC 7662 endpoint immediately after the OAuth callback to confirm the user's access token is active. The result appears as user-token-introspection in the Token Chain panel and shows the live sub, scope, exp, and aud from PingOne.
  • At every MCP tool call — demo_mcp_gateway (port 3005) performs RFC 7662 introspection (Step 1 of its inline authorization pipeline) before forwarding the call. The BFF performs the RFC 8693 exchange to produce the bearer token; the gateway then introspects that token against PingOne before executing the tool.

All other tokens in the chain (agent CC token, exchanged MCP token — and, when the optional two-exchange path is enabled, intermediate actor tokens) use RFC 7515 (JWS) cryptographic signature verification instead — faster because no network round-trip is needed, and appropriate because these tokens were just issued by PingOne moments ago and cannot have been revoked in that window.
