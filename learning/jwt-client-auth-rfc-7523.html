What is it

JWT-based client authentication — RFC 7523 §3 — lets an OAuth client prove its identity to the authorization server using a signed JWT instead of a shared secret (client_secret). This is also calledprivate_key_jwt (for asymmetric keys) orclient_secret_jwt (for HMAC).

Instead of sending client_id + client_secret, the client generates a JWT assertion, signs it with its private key, and sends it asclient_assertion_type + client_assertion to the token endpoint:

POST /as/token Content-Type: application/x-www-form-urlencoded grant_type=authorization_code &code=SplxlOBeZQQYbYS6WxSbIA &redirect_uri=https://app.example.com/callback &client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer &client_assertion=eyJhbGciOiJSUzI1NiIsImtpZCI6Im15LWtleSJ9.eyJpc3MiOiJjbGllbn…

JWT assertion structure

Required claims in the client assertion JWT (RFC 7523 §3)

ClaimValue
issclient_id — the issuer is the client itself
subclient_id — same as iss for client authentication
audToken endpoint URL of the AS (e.g. https://auth.pingone.com/{envId}/as/token)
jtiUnique ID — prevents replay (AS must track and reject reuse)
expExpiry — short-lived, typically now + 60–120 seconds
iatIssued at time

Example decoded header + payload:

// Header { "alg": "RS256", "kid": "my-signing-key-2026" } // Payload { "iss": "my-bff-client-id", "sub": "my-bff-client-id", "aud": "https://auth.pingone.com/env-123/as/token", "jti": "a87fec1a-3b9d-4c2e-a12f-7bcd9ef01234", "iat": 1711617600, "exp": 1711617720 }

In token exchange

RFC 7523 in RFC 8693 Token Exchange

In this demo, the Backend-for-Frontend (BFF) performs an RFC 8693 token exchange to mint an MCP token. The BFF authenticates itself to the PingOne token endpoint using private_key_jwt — proving it is the authorised actor (act) delivering the delegation chain:

POST /as/token grant_type=urn:ietf:params:oauth:grant-type:token-exchange &subject_token=<user-access-token> &subject_token_type=urn:ietf:params:oauth:token-type:access_token &requested_token_type=urn:ietf:params:oauth:token-type:access_token &audience=banking_mcp_server &scope=read &client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer &client_assertion=<signed-jwt>

The resulting MCP token carries both sub (the end-user) andact.client_id (the BFF), proving the delegation chain without transmitting any long-lived shared secret.

PingOne setup

Enabling private_key_jwt in PingOne

  1. In the PingOne admin console, open your Application → Configuration.
  2. Under Token Endpoint Authentication Method, selectprivate_key_jwt or client_secret_jwt.
  3. For private_key_jwt: provide your client's JWKS URI or upload the public key. PingOne will fetch the JWKS to verify incoming assertion signatures.
  4. Set the Token Endpoint Authentication Signing Algorithm(RS256, RS384, ES256, PS256 …). Must match your signing key type.
  5. In your BFF: generate a signed JWT with the required claims (see "JWT assertion structure" tab) and include it as client_assertion on every token endpoint call.

PingOne's metadata document advertises supported methods undertoken_endpoint_auth_methods_supported in/.well-known/openid-configuration.
