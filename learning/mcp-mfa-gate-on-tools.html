How it works

What powers the AI assistant?

The AI assistant in this app doesn't just generate text — it connects to the banking system to fetch real data and perform real actions. The technology that makes this possible is called MCP (Model Context Protocol).

Think of MCP like a universal remote control. Just as one remote can control your TV, sound bar, and streaming box using a standard set of buttons, MCP gives any AI assistant a standard way to "press the buttons" on any connected service — whether that's a bank, a calendar, or a database.

This means the AI doesn't need custom code for every banking feature. Instead, the banking system publishes a tool catalog (a menu of things it can do), and the AI picks from that menu when responding to your requests.

How a tool call works

When you ask the AI "what's my balance?" here's what happens under the hood:

1. You type a question
2. The AI understands you want "get_account_balance"
3. The app sends a secure, signed request to the Banking Service:
     → "call get_account_balance for account acc_001"
4. The Banking Service checks your identity and permissions
5. It returns the result to the AI
6. The AI formats a friendly response for you

The AI never has direct access to your account data. Every single tool call goes through the same security checks as if you had clicked a button in the app.

What tools are available?

The Banking Service currently provides these AI-callable tools:

  • Get my accounts — lists all your accounts
  • Check balance — balance for a specific account
  • Recent transactions — your latest activity
  • Transfer funds — move money between accounts
  • Deposit — add funds to an account
  • Withdraw — take funds from an account

How the AI knows what tools exist

When the AI assistant starts up, it asks the Banking Service: "What can you do?" The service responds with a tool catalog — each tool has a name, a description, and the parameters it needs. The AI reads this catalog and figures out which tool to call when you make a request.

AI → Banking Service: "What tools do you have?"
Banking Service → AI:
  {
    "tools": [
      { "name": "get_my_accounts", "description": "List all accounts for the authenticated user" },
      { "name": "create_transfer",  "description": "Transfer funds between accounts" },
      ...
    ]
  }

Security: every tool call is verified

The AI cannot just call any tool it wants. Every call is accompanied by a security pass (an OAuth access token) that proves: ✅ who you are, ✅ what you're allowed to do, and ✅ that the AI is acting on your behalf. The Banking Service checks this pass before executing anything.

Server discovery

How external agents find this server

The /.well-known/mcp-server endpoint is a public discovery manifest that lists all available tools and their auth requirements. External MCP clients — Claude Desktop, Cursor, Windsurf — can query it before connecting to understand what the server offers and which tools require authentication.

Discovery endpoint

GET {MCP_SERVER_URL}/.well-known/mcp-server

Example response (tool access tiers)

{
  "name": "super-banking",
  "tools": [
    { "name": "get_my_accounts", "readOnly": true, ... },
    { "name": "create_transfer", "readOnly": false, ... }
  ],
  "publicAccess": {
    "readOnlyTools": [
      "get_my_accounts",
      "get_account_balance",
      "get_my_transactions",
      "sequential_think"
    ]
  },
  "restrictedAccess": {
    "authenticatedTools": [
      "get_sensitive_account_details",
      "create_deposit",
      "create_withdrawal",
      "create_transfer",
      "query_user_by_email"
    ]
  },
  "auth": {
    "type": "oauth2",
    "authorizationUrl": "https://auth.pingone.com/.../authorize",
    "tokenUrl": "https://auth.pingone.com/.../token"
  }
}

The publicAccess.readOnlyTools list is what model-context–aware AI clients use to offer a preview of capabilities before asking the user to authenticate.

Two AI paths

Two ways the AI can connect

Web app path (you are here): when you use the banking assistant in this browser, your session cookie is used to verify who you are. The app exchanges this for a secure, limited-access AI tools pass before calling any banking functions. Your browser never holds a security token.

Autonomous AI agent path: a separate AI process (powered by an external reasoning service) can also connect to the same banking tools. Instead of your browser session, it uses its own machine credentials and can request approval from you out-of-band — for example, via an email or push notification — before performing high-value actions.

Both paths ultimately hit the same Banking Service and are subject to the same security checks. The difference is how they prove identity and get authorized.

Auth challenge

What is an auth challenge?

An auth challenge occurs when an MCP tool call requires additional authorization before it can proceed. Rather than silently failing, the server signals the need for user approval so the flow can complete after authorization is granted.

Three trigger scenarios in this demo

  1. Amount threshold exceeded — transfers above the configured CIBA threshold trigger a push notification to the user out-of-band. The agent waits and retries when approval arrives.
  2. PingOne Authorize gate — high-risk tools (e.g., large withdrawals) are checked against a policy decision point before the token exchange proceeds.
  3. Mid-flow session loss — if the user’s session expires during an agent operation, an inline login prompt appears in the browser. After re-authentication, the agent continues automatically.

Flow sequence

User clicks "Transfer $10,000" in agent
     ↓
BFF POST /api/mcp/tool → mcpToolAuthorizationService
     ↓
PingOne Authorize gate checks RAR / scopes
     ↓  [step-up needed]
BFF returns { error: "step_up_required", cibaRequired: true }
     ↓
UI shows "Approval needed" — polls /api/ciba/status every 5s
     ↓  [user approves on device / email link]
BFF retries tool call with elevated token
     ↓
Tool executes; result returned to agent

Why this pattern matters

Agent operations get consent while keeping the user in control. The token is never escalated silently — every step-up is user-visible and audit-logged. This aligns with HITL (Human-in-the-loop) best practices and the FAPI 2.0 security profile for high-value transactions.

Note: “All token exchange happens inside the BFF (server-side). The browser only holds a session cookie.”

MFA gate on tools

MFA gate on tool discovery

GET /api/mcp/inspector/tools (tools/list) requires an authenticated session. Unauthenticated requests receive 401 Unauthorized, preventing tool enumeration by unauthenticated callers.

GET /api/mcp/inspector/tools?sessionId=...

If no valid session:
  → 401 { error: "unauthorized", message: "Login required" }

If session present with valid access token:
  → 200 { tools: [ { name, description, inputSchema }, ... ] }

BFF validation steps:
  1. Session cookie present and not expired
  2. session.oauthTokens.accessToken valid (or refreshed via refresh_token)
  3. Token forwarded to MCP server tools/list call

Why gate tools/list?

An unauthenticated caller could enumerate all available banking operations — account balance, transfer funds, device authentication — without being a real user. Gating the tool list at the BFF means even tool discovery is behind the same auth boundary as tool execution.

Step-up MFA for high-value tool calls

POST /api/mcp/tool (tools/call) for transactions at or above the configured threshold triggers step-up authentication before the tool executes. The agent receives a 401 with step_up_required: true and must wait for the user to complete MFA before retrying.

POST /api/mcp/tool { tool: "transfer_funds", amount: 5000 }

Below threshold:
  → 200 { result: "Transfer complete" }

At/above threshold (step-up):
  → 401 { step_up_required: true, cibaAuthId: "..." }
  UI shows MFA prompt; polls /api/ciba/status
  On approval → retry POST /api/mcp/tool (same body)
  → 200 { result: "Transfer complete" }

Handshake sequence

MCP JSON-RPC Handshake

Every MCP session opens with a fixed four-step handshake over WebSocket. Messages are JSON-RPC 2.0. MCP 2025-11-25

Step 1 — Client → Server: initialize

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-11-25",
    "capabilities": { "tools": {} },
    "clientInfo": { "name": "super-banking-bff", "version": "1.0.0" }
  }
}

Step 2 — Server → Client: initialize response

{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-11-25",
    "capabilities": { "tools": { "listChanged": false } },
    "serverInfo": { "name": "super-banking-mcp", "version": "1.0.0" }
  }
}

Step 3 — Client → Server: notifications/initialized

Notification only — no response expected from server.

{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}

Step 4 — Client → Server: tools/list

// Request
{ "jsonrpc": "2.0", "id": 2, "method": "tools/list", "params": {} }

// Response (excerpt)
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "tools": [
      {
        "name": "get_my_accounts",
        "description": "List accounts for the authenticated user",
        "inputSchema": { "type": "object", "properties": {}, "required": [] }
      }
    ]
  }
}

Calling a tool — tools/call

// Request
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "get_my_accounts",
    "arguments": {}
  }
}

// Success response
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [{ "type": "text", "text": "[{...}]" }],
    "isError": false
  }
}

// Tool execution error (NOT a protocol error — still HTTP 200)
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [{ "type": "text", "text": "Insufficient funds" }],
    "isError": true
  }
}

Tool list change notification

// Server pushes when available tools change (e.g. after auth state change)
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed"
}
// Client should re-issue tools/list after receiving this

Streamable HTTP transport — key headers

Modern MCP uses Streamable HTTP (POST + optional SSE) instead of a persistent WebSocket.

// Client must accept both JSON and SSE
Accept: application/json, text/event-stream

// Version negotiation header (per-request in modern MCP)
MCP-Protocol-Version: 2025-11-25

// Session tracking (server assigns on first response)
Mcp-Session-Id: 1868a90c...

// Resume a broken SSE stream
GET /mcp HTTP/1.1
Last-Event-ID: <last-seen-event-id>
  • Client POSTs every request; server replies with JSON or opens SSE stream
  • Mcp-Session-Id must be echoed on all subsequent requests
  • Disconnection is NOT a cancellation — use CancelledNotification to cancel
  • Server returns 404 when session expires; client must re-initialize
The BFF performs the MCP handshake server-side and proxies tool calls viaPOST /api/mcp/tool. The browser never opens a WebSocket directly — all token exchange and session management stays in the BFF.
